# 소스 다층화 조사 반영 가능성 검토 보고서

작성일: 2026-06-23

## 1. 결론

첨부 조사 내용은 현재 프로그램에 상당히 많이 반영할 수 있다. 특히 핵심 방향은 맞다.

현재까지 만든 빅카인즈 공개화면 기반 방향성 기능은 “수치가 글의 방향을 정하는 문제”를 줄이는 1차 개선이다. 다만 조사 내용처럼 빅카인즈 하나에 의존하면 장기 운영 안정성이 낮다. 다음 단계는 빅카인즈를 더 강하게 긁는 것이 아니라, 여러 소스를 같은 구조로 받아 “방향성 신호”와 “사실 검증 출처”를 분리하는 것이다.

추천 구조는 아래 한 줄로 정리된다.

```text
빅카인즈로 뉴스 의제를 잡고, 네이버/공공데이터/RSS/GDELT로 대체 신호를 만들고, 공식기관 데이터로 사실을 고정한다.
```

즉, 프로그램은 단순 수집기가 아니라 “편집장 엔진”으로 가야 한다.

## 2. 현재 코드에 이미 반영된 부분

### 2.1 빅카인즈를 방향성 신호로 쓰는 구조

이미 `BigKindsPublicCollector`가 추가되어 공개 화면에서 방향성 후보를 수집하는 구조가 들어갔다.

- 파일: `modules/collectors/bigkinds_public.py`
- 수집 대상: 이슈명, 카테고리, 키워드, 기사 건수, 공개 링크, 수집 시각, 신뢰도
- 제외 대상: 기사 본문, 로그인 필요 데이터, 전문 복제
- 운영 안전장치: 캐시, 실패 시 빈 결과 반환, 본문 저장 금지 테스트

이 부분은 조사 내용의 “빅카인즈는 방향성 신호로 쓰되, 본문 크롤링 중심은 피하라”는 방향과 일치한다.

### 2.2 수치보다 화자의 목적을 먼저 세우는 플래너

`DirectionalTopicPlanner`가 추가되어 `confirmed_metrics`를 글감의 출발점이 아니라 근거 역할로 바꾸는 흐름이 들어갔다.

- 파일: `modules/market/directional_topic_planner.py`
- 출력: `reader_problem`, `speaker_purpose`, `angle`, `title_candidates`, `evidence_roles`
- 핵심 변화: 금리, 환율, 지수, 코인 수치를 제목 앞에 세우지 않고 본문 근거로 배치

이건 첨부 조사에서 제안한 `article_angle.json`의 초기 버전으로 볼 수 있다. 현재 이름은 `EditorialIntent`지만 역할은 거의 같다.

### 2.3 생성 파이프라인 연결

콘텐츠 생성 전 단계에서 방향성 컨텍스트가 먼저 들어가도록 연결되어 있다.

- 파일: `modules/llm/content_generator.py`
- 동작: 시장 스냅샷과 `confirmed_metrics`를 읽고, 빅카인즈 이슈와 결합해 “방향성 주제 플랜”을 프롬프트 앞단에 삽입
- 결과: `seo_snapshot["editorial_intent"]`에 방향성 의도가 남는다.

### 2.4 제목 품질 게이트

수치 나열형 제목으로 되돌아가는 것을 막는 게이트가 들어갔다.

- 파일: `modules/automation/publication_preflight.py`
- 경고 코드: `DIRECTIONAL_TITLE_WEAK`
- 기본 상태: 경고
- 강화 옵션: `AUTOBLOG_DIRECTIONAL_TITLE_REQUIRED=true`이면 차단 가능

### 2.5 텔레그램 승인 메시지 개선

텔레그램 승인 요청에 방향성과 수치 근거가 분리되어 표시되도록 개선되어 있다.

- 파일: `modules/automation/draft_approval.py`
- 표시 항목: 방향성 주제, 화자 목적, 근거 수치
- 효과: 사용자가 “이 글이 왜 이 주제로 생성됐는지”를 승인 단계에서 볼 수 있다.

## 4. 바로 반영하면 좋은 설계

### 4.1 `DirectionSignal` 공통 스키마

현재는 빅카인즈 이슈가 `BigKindsIssue`로 들어오고, 네이버/RSS/GDELT는 각자 다른 형태로 흐른다. 다음 단계에서는 모든 방향성 후보를 공통 스키마로 바꾸는 것이 좋다.

추천 필드:

```json
{
  "signal_id": "bigkinds:2026-06-23:ai-semiconductor",
  "source": "BigKinds public",
  "source_tier": "agenda",
  "title": "AI 반도체 투자 확대",
  "keywords": ["AI", "반도체", "전력", "데이터센터"],
  "entities": ["삼성전자", "엔비디아"],
  "category": "경제",
  "count": 42,
  "trend_score": 0.78,
  "audience_score": 0.61,
  "authority_score": 0.7,
  "market_relevance": 0.84,
  "risk_flags": [],
  "url": "https://www.kinds.or.kr/",
  "collected_at": "2026-06-23T20:30:00+09:00"
}
```

이렇게 만들면 빅카인즈, 네이버 뉴스, 네이버 블로그, RSS, GDELT를 한 점수판에서 비교할 수 있다.

### 4.2 방향성 점수화

첨부 조사에서 제안한 점수화는 그대로 쓸 가치가 있다. 다만 현재 코드 기준으로는 아래처럼 약간 조정하는 편이 현실적이다.

```text
Direction Score =
뉴스/의제 강도 25%
검색/블로그 관심도 20%
시장 수치 연결성 20%
공식 출처 연결성 15%
블로그 적합도 10%
사용자 관심 키워드 10%
```

현재 프로그램에서는 `confirmed_metrics`와 `source_pack`이 이미 있으므로 “시장 수치 연결성”과 “공식 출처 연결성”은 바로 계산 가능하다. 네이버 API 키가 있으면 “검색/블로그 관심도”도 바로 붙일 수 있다.

### 4.3 `ArticleAngle` 확장

현재 `EditorialIntent`는 방향성 의도의 최소 구현이다. 다음 단계에서는 조사 내용의 `article_angle.json`에 맞춰 확장하는 것이 좋다.

추천 확장 필드:

```json
{
  "main_topic": "AI 반도체 투자 확대",
  "angle": "반도체를 맞히기보다 실제 수요가 숫자로 확인되는지를 보자",
  "why_today": [
    "뉴스 의제 강도가 높음",
    "국내 시장과 반도체 키워드 연결성이 높음",
    "금리와 환율이 성장주 부담을 확인해야 하는 구간"
  ],
  "article_type": "투자 원칙형",
  "supporting_sources": ["BigKinds public", "Naver news", "BOJ", "FRED"],
  "do_not_claim": [
    "특정 종목 매수를 권유하지 않는다",
    "단일 뉴스로 시장 방향을 단정하지 않는다"
  ]
}
```

현재 `EditorialIntent`를 버릴 필요는 없다. 먼저 `EditorialIntent`에 `why_today`, `article_type`, `do_not_claim`, `supporting_sources`를 추가하면 된다.

## 5. 추천 구현 순서

### 1단계: 방향성 신호 통합기

가장 먼저 할 일은 새 크롤러를 많이 붙이는 것이 아니라 `DirectionSignalAggregator`를 만드는 것이다.

작업:

- `modules/market/direction_signal.py` 추가
- `DirectionSignal` 데이터 모델 추가
- `DirectionSignalAggregator` 추가
- `BigKindsIssue`를 `DirectionSignal`로 변환
- 점수 계산과 fallback 사유 기록

기대 효과:

- 빅카인즈가 실패해도 네이버/RSS/GDELT 신호를 같은 방식으로 비교 가능
- “왜 이 주제가 선택됐는지”를 로그와 텔레그램에 설명 가능

### 2단계: 네이버 뉴스/블로그 관심도 연결

이미 `modules/collectors/naver_search.py`가 있으므로 구현 난도가 낮다.

작업:

- 빅카인즈 후보 키워드로 네이버 뉴스 검색
- 같은 키워드로 네이버 블로그 검색
- 뉴스 결과 수와 블로그 결과 수, 제목 반복 키워드 비교
- `audience_score` 계산

예상 개선:

- 언론 의제와 실제 블로그 독자 관심도의 차이를 반영
- 제목이 더 블로그 친화적으로 바뀜
- 예: “환율 상승”에서 “달러 환전 타이밍을 어떻게 볼까” 같은 실사용 관점 도출

필요 조건:

- `NAVER_CLIENT_ID`
- `NAVER_CLIENT_SECRET`

### 3단계: RSS/GDELT 방향성 승격

RSS와 GDELT는 이미 시장 스냅샷 수집에 쓰이고 있다. 이것을 “뉴스 컨텍스트”에서 “방향성 후보”로 승격시키면 된다.

작업:

- RSS 뉴스 제목/요약에서 반복 키워드 추출
- GDELT 기사 제목/도메인/발행시각을 `DirectionSignal`로 변환
- 해외 이슈는 GDELT 가중치 상승
- 국내 국장전 글은 BigKinds/Naver 가중치 상승

예상 개선:

- 중국, 일본, 미국장, 반도체, AI, 중동 이슈 확장에 유리
- 빅카인즈 실패 시에도 주제 선정 가능

### 4단계: 공공데이터 빅카인즈 Top10 검증 후 추가

조사 내용상 공공데이터포털 빅카인즈 Top10은 매우 좋은 fallback이다. 다만 구현 전 확인이 필요하다.

확인할 것:

- 실제 API가 서비스키를 요구하는지
- 파일 다운로드 방식인지
- 일별 업데이트 시각
- 응답 필드
- 상업적 블로그 운영에서 사용 가능한지

구현 판단:

- 서비스키가 필요하면 사용자 발급 후 추가
- 파일데이터가 공개 URL로 안정 제공되면 keyless fallback으로 추가
- API 제한이 크면 하루 1회 캐시 방식으로 운용

### 5단계: KPF-BERT/NER 장기 적용

KPF BIGKINDS-LAB는 장기적으로 의미가 있다. 다만 바로 넣기보다 데이터가 쌓인 뒤 도입하는 것이 좋다.

적용 방식:

- 1차: 제목/요약 기반 키워드 규칙과 간단한 개체명 추출
- 2차: KPF-BERT 뉴스분류 모델 적용
- 3차: KPF NER로 기업, 기관, 인물, 지역 추출
- 4차: 자체 이슈 클러스터 생성

기대 효과:

- 빅카인즈가 막혀도 자체적으로 이슈 묶음 생성
- 국내 기사 제목과 요약을 경제/산업/정책/국제 이슈로 자동 분류

주의:

- 모델 의존성, 실행 속도, 로컬 메모리 부담 확인 필요
- 처음부터 모델을 넣으면 운영 복잡도가 커진다.

## 6. API 키 없이 가능한 것과 필요한 것

### API 키 없이 우선 가능한 것

- 빅카인즈 공개화면 저빈도 수집
- 공식기관 RSS
- 구글뉴스 RSS
- GDELT DOC API
- BOJ 공개 데이터
- 중국 NBS 공개 링크 기반 컨텍스트
- DART/KIND의 공개 RSS 또는 공개 화면 일부
- 기존 시장 수치 기반 `confirmed_metrics`

### API 키가 있으면 품질이 크게 좋아지는 것

- 네이버 뉴스 API
- 네이버 블로그 API
- 공공데이터포털 빅카인즈 Top10 API
- OpenDART API
- YouTube Data API
- NewsAPI/Mediastack/Media Cloud

### 비공식이라 보조로만 둘 것

- pytrends
- 공개 사이트 구조에 강하게 의존하는 크롤러
- 네이버 뉴스 본문 크롤러
- 기사 전문 추출 라이브러리

## 7. 운영 리스크 판단

### 낮은 리스크

- 공식 API
- 공식 RSS
- 공개 데이터 API
- 제목/요약/메타데이터 중심 수집
- 링크를 남기고 원문 전문을 저장하지 않는 방식

### 중간 리스크

- 공개 화면 저빈도 수집
- Google News RSS
- GDELT 기반 글로벌 뉴스 수집
- RSSHub 자체 호스팅

### 높은 리스크

- 로그인 페이지 수집
- 캡차/차단 우회
- 뉴스 본문 대량 저장
- 네이버 기사 본문 크롤링 중심 운영
- 구조 변경이 잦은 화면을 핵심 의존점으로 삼는 방식

## 8. 현재 프로그램에 대한 최종 추천

가장 현실적인 다음 코딩 순서는 아래와 같다.

1. `DirectionSignal` 공통 모델과 `DirectionSignalAggregator`를 만든다.
2. 기존 `BigKindsIssue`를 `DirectionSignal`로 변환한다.
3. 네이버 뉴스/블로그 API 결과를 방향성 점수에 반영한다.
4. 기존 RSS/GDELT 결과를 방향성 후보로 승격한다.
5. `EditorialIntent`를 `ArticleAngle`에 가깝게 확장한다.
6. 텔레그램 승인 메시지에 “선택된 소스 계층”과 “fallback 여부”를 표시한다.
7. 공공데이터 빅카인즈 Top10은 API/파일 제공 방식 확인 후 추가한다.
8. KPF-BERT/NER는 데이터가 쌓인 뒤 장기 과제로 적용한다.

## 9. 기대되는 글 품질 개선

이 구조가 들어가면 글의 품질은 아래 방향으로 좋아진다.

- 제목이 “금리, 환율, 지수” 나열에서 벗어난다.
- 글의 첫 문장이 “오늘 무엇을 판단해야 하는가”로 시작된다.
- 수치는 방향을 정하는 재료가 아니라 판단을 보강하는 증거가 된다.
- 언론 의제와 블로그 독자 관심도의 차이를 반영할 수 있다.
- 빅카인즈가 실패해도 RSS/GDELT/네이버로 축소 운행 가능하다.
- 국내, 중국, 일본, 미국 이슈를 같은 구조로 비교할 수 있다.
- 텔레그램 승인 단계에서 “왜 이 주제가 선택됐는지”가 더 직관적으로 보인다.

## 10. 의사결정 제안

바로 코딩할 다음 작업은 `DirectionSignalAggregator`가 좋다. 이유는 새 소스를 더 붙이기 전에 받을 그릇을 먼저 만들어야 하기 때문이다.

그 다음은 네이버 뉴스/블로그 방향성 점수 연결이 좋다. 이미 `NaverSearchCollector`가 있어서 구현 난도가 낮고, 블로그 글 품질에 직접적으로 영향을 준다.

공공데이터 빅카인즈 Top10은 매우 유용하지만, 실제 API 키 필요 여부와 응답 구조를 확인한 뒤 구현하는 것이 맞다. KPF-BERT/NER는 장기적으로 좋지만 지금 바로 넣기에는 운영 복잡도가 크므로 2차 고도화 과제로 두는 것이 좋다.
