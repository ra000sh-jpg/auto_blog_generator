# 네이버 블로그 자동화 프로그램 점검 및 재구성 계획

작성일: 2026-06-05
목표: CSV 주제 입력을 바탕으로 하루 3개 수준의 고품질 네이버 블로그 초안을 만들고, 텔레그램 승인 후 네이버 블로그 임시저장까지 자동화한다.

## 1. 결론

현재 프로그램은 단순 스크립트가 아니라 대시보드, 스케줄러, 텔레그램, 멀티 LLM 라우터, 이미지 생성/검색, 품질 게이트, 네이버 Playwright 업로더, 메모리 저장소까지 포함한 플랫폼형 구조다.

기능 자산은 많지만, 사용자가 원하는 목표인 "가볍고 블로그만을 타겟한 고품질 글 양산" 관점에서는 아직 운영 완성 상태가 아니다. 특히 네이버 업로더가 현재는 최종 발행 중심이고, 목표인 임시저장 모드가 별도 인터페이스로 분리되어 있지 않다. 또한 실제 DB에 남아 있는 준비 초안 일부는 제목과 본문 관점이 어긋나거나, "자동 đăng재", "涉해"처럼 한국어 글 품질을 해치는 혼입 표현이 보여 바로 양산하기에는 위험하다.

따라서 권장 방향은 "새로 만들기"가 아니라, 기존 자산 중 필요한 것만 남겨 블로그 전용 경량 파이프라인으로 재구성하는 것이다.

## 2. 사용자의 목표 반영

- 최종 자동화 목표: 네이버 블로그 임시저장까지.
- 업로드 방식: 브라우저 자동화 허용.
- 운영량: 하루 3개.
- 입력: CSV 파일.
- 글 연결성: 앞으로 생성될 블로그 글 백업과 메모리를 기반으로 연결.
- 페르소나: "윤서재"형 현실 검증 블로거. 40대 초반 남성, 프랜차이즈 카페 운영 경험, INTJ 성향, 생활형 사업가 + 장기투자자 + AI 실험가.
- 타깃 독자: 투자를 공부하기 시작한 초심자, 자기개발 관심자.
- 글 성격: 정보 전달 + 통찰 + 철학적 견해.
- 이미지: AI 이미지 생성과 무료 이미지 자동검색 혼합.
- 승인 방식: 텔레그램으로 텍스트 초안을 먼저 받고, 답장/버튼 승인 후 네이버 임시저장.
- 서버 비용: 월 3달러 이하.

## 3. 검증 결과

### 통과한 항목

- `python3 -m compileall -q modules server scripts tools test_generation.py test_db.py` 통과.
- `frontend`의 `npm run lint` 통과. 경고는 `telegram-connect-card.tsx`의 `<img>` 최적화 경고 1건뿐이다.
- 기존 코드에는 생성, 품질 검사, 이미지 생성/검색, 마크다운 변환, 네이버 에디터 입력, 스케줄러, 텔레그램, 준비 초안 저장 구조가 모두 존재한다.
- `data/automation.db` 기준 `topic_memory`가 61건 있어 과거 발행 이력 기반 연결성 기능의 토대가 있다.
- `ready_to_publish`, `awaiting_images`, `prepared_payload` 구조가 있어 "생성 후 나중에 업로드" 흐름을 재사용할 수 있다.

### 차단 또는 실패한 항목

- 시스템 Python에 `pytest`, `PyYAML`, `requests` 등이 없어 전체 pytest는 실행하지 못했다.
- `modules.automation` import 시 `trend_job_service`와 collector가 eager import되어 `requests`가 없으면 파이프라인 import가 실패한다. 블로그 전용 경량화에서는 선택 기능 import를 늦춰야 한다.
- `npm run build`는 Next.js SWC 바이너리가 `node_modules/@next/swc-darwin-arm64`에 실제 `.node` 파일 없이 메타 파일만 있어 실패했다.
- 현재 DB 설정은 `scheduler_daily_posts_target=4`, `telegram_image_mode=auto`로 되어 있어 사용자의 목표인 하루 3개, 텔레그램 승인 중심과 다르다.
- 현재 DB에는 `failed=13`, `completed=10`, `awaiting_images=7`, `ready_to_publish=3`, `queued=2`, `running=1` 상태가 남아 있다. 운영 큐 정리가 필요하다.

## 4. 콘텐츠 품질 평가

현재 프롬프트 구조는 상당히 고도화되어 있다. `modules/llm/prompts.py`에는 인지적 깊이, 반AI 패턴, 문장 다듬기, 페르소나, 토픽 모드가 들어 있다. 하지만 실제 준비 초안 샘플에서는 아래 문제가 보였다.

- 제목과 본문 관점이 틀어짐: "연 매출 3억 8천만 원 빽다방 운영..." 제목인데 본문 제목이 "다양한 생각들로 성공을 위한 전략 수립"으로 희석됨.
- 어색한 다국어 혼입: "자동 đăng재", "涉해" 같은 표현.
- 일반론 과잉: "다양한 생각들이 필요하다", "새로운 기회를 창출한다"처럼 글쓴이만의 판단 기준이 약함.
- 페르소나 불일치: 현재 코드의 고정 캐릭터는 "카페사장 아빠"에 가깝고, 첨부 페르소나의 "윤서재 / 생활형 분석 블로그 / 투자 초심자 대상"과 완전히 일치하지 않는다.
- 품질 상태 혼선: 일부 실패 잡은 `error_code=QUALITY_FAILED`인데 `quality_snapshot.score=85`와 `gate=pass`가 같이 저장되어 있다. LLM 품질 게이트와 로컬 품질 게이트가 다른 결론을 낸 것으로 보이며, 운영자가 원인을 해석하기 어렵다.

판정: 현재 생성 엔진은 구조적으로 강하지만, 실제 양산 품질은 아직 "검수 없이 임시저장"하기 어렵다. 먼저 페르소나와 품질 게이트를 블로그 전용으로 다시 고정해야 한다.

## 5. 네이버 자동화 및 GitHub 사례 분석

네이버 개발자센터 공지에 따르면 로그인 방식 블로그 Open API 글쓰기 기능은 2020년 5월 6일 종료되었다. 공지 이유에는 반복적, 기계적, 유사 글 대량 발행 문제가 언급되어 있다. 따라서 공식 글쓰기 API 기반 자동화는 현실적 선택지가 아니며, 브라우저 자동화가 사실상 유일한 경로다.

공개 GitHub/Gist 사례들도 같은 결론이다. Selenium 기반 자동화 예시는 SmartEditor ONE이 React 기반이며, 셀렉터 변동, iframe, 발행 버튼 클릭 실패, 캡차, headless 환경 문제, Xvfb/클립보드 폴백을 핵심 이슈로 다룬다. 현재 프로그램의 Playwright 방식은 이보다 현대적이고, 세션 저장, stealth, 팝업 취소, 이미지 업로드, AI 활용 설정까지 이미 상당 부분 대응하고 있다.

다만 공개 사례와 비교했을 때 현재 프로그램에 꼭 추가해야 하는 것은 "발행"과 "임시저장"을 분리하는 명시적 인터페이스다. 현재 `PlaywrightPublisher.publish()`는 2차 발행 버튼까지 누르는 구조라서 사용자의 1차 목표와 다르다.

## 6. 보존할 자산과 줄일 자산

### 보존 권장

- `JobStore`: SQLite 기반 큐, 상태, prepared payload, topic memory 기반으로 유지.
- `PipelineService`: 생성/이미지/업로드 오케스트레이션은 유지하되 블로그 전용으로 축소.
- `ContentGenerator`: 품질 레이어와 Voice 레이어 아이디어는 좋다. 페르소나만 재정의.
- `ImageGenerator`: AI/스톡 혼합 전략은 목표와 맞다.
- `TelegramNotifier` 및 webhook: 승인 플로우의 기반으로 사용.
- `PlaywrightPublisher`와 `NaverEditorHelper`: 네이버 에디터 대응 자산으로 유지.
- 스케줄러의 자연스러운 발행 슬롯: 하루 3개 운영에 적합.
- `topic_memory`: 향후 연계성 확보에 핵심.

### 축소 또는 선택 기능화 권장

- 멀티채널 발행: 네이버 블로그 전용 목표에서는 기본 비활성화.
- WordPress/Tistory Publisher: 지금은 불필요.
- VLM 모델 자동 발견/가격 검증/챔피언 경쟁: 비용과 복잡도가 크다. 글 품질이 안정된 뒤 선택 기능으로 둔다.
- 무거운 대시보드: 운영에 필요한 최소 화면만 남긴다.
- 과도한 모델 라우팅: 월 3달러 이하 목표에서는 기본 모델 조합을 고정하고, 실패 시에만 폴백한다.
- 트렌드/RSS 자동 수집: 처음에는 CSV 입력 기반으로 제한한다.

## 7. 목표 아키텍처

권장 파이프라인:

1. CSV 입력
   - `title`, `keywords`, `category`, `topic_mode`, `priority`, `notes` 컬럼.
   - ChatGPT가 100개 제목을 CSV로 만들 수 있도록 템플릿 제공.

2. 주제 사전검사
   - 기존 `topic_memory`와 유사도 비교.
   - 너무 비슷하면 "후속 글", "비교 글", "실험 기록"으로 각도를 바꾼다.

3. 윤서재 페르소나 생성
   - 첨부 페르소나를 `prompts.py` 또는 DB persona profile에 고정.
   - 기본 구조는 "왜 다루는가 → 처음 생각 → 확인한 내용 → 장점 → 리스크 → 맞는 사람 → 맞지 않는 사람 → 결론 → 후속 글".

4. 품질 게이트
   - 금지 표현 검사.
   - 제목-본문 정합성 검사.
   - 한국어 혼입 오류 검사.
   - 투자 글 고지: 투자 권유가 아니라 판단 기준 공유.
   - 페르소나 일치도 검사.

5. 이미지 계획
   - 글당 기본 2~3장.
   - 투자/AI/생산성: AI 이미지 1장 + 무료 스톡 1~2장.
   - 실제 제품/장소/생활 주제: 스톡 우선.
   - 이미지 없는 글도 허용하되, 네이버 첫 화면용 썸네일은 기본 생성.

6. 텔레그램 초안 승인
   - 텍스트 초안 요약 + 전체 보기 링크 또는 파일.
   - 버튼: `승인`, `수정 요청`, `폐기`, `나중에`.
   - 승인 시 `approved_for_draft` 상태로 전환.

7. 네이버 임시저장
   - `publish()`와 별도로 `save_draft()` 추가.
   - 임시저장 성공 시 `draft_saved` 상태와 네이버 편집 URL/스크린샷 저장.
   - 최종 발행은 별도 수동 또는 2단계 자동 승인으로 분리.

8. 백업 및 메모리
   - 임시저장 payload와 최종 발행 URL을 모두 저장.
   - 다음 글 생성 시 과거 제목, 관점, URL, 핵심 결론을 프롬프트에 주입.

## 8. 배포 전략

### 1단계: 로컬 Mac LaunchAgent

가장 현실적인 첫 운영 방식이다. 네이버 로그인 세션, 캡차, 브라우저 프로필 유지가 쉽고, 현재 저장소에도 LaunchAgent 스크립트가 있다. 하루 3개 초안 생성/임시저장까지는 로컬 상시 운영이 가장 안정적이다.

### 2단계: Google Cloud Run Jobs

Cloud Run은 공식적으로 Playwright 같은 브라우저 자동화를 컨테이너에서 실행하는 방식을 문서화하고 있다. Jobs는 HTTP 서버가 아니라 실행 후 종료하는 작업에 적합하다. Cloud Scheduler와 연결하면 하루 몇 번 실행하는 구조를 만들 수 있다.

단점은 네이버 로그인 세션/캡차/브라우저 지문 관리가 로컬보다 어렵다는 점이다. 따라서 Cloud Run은 "생성 + 텔레그램 승인"까지 먼저 올리고, 네이버 임시저장은 로컬 또는 별도 브라우저 실행기로 유지하는 하이브리드가 안전하다.

### GitHub Actions

GitHub Actions는 스케줄 실행과 Secrets에는 좋지만, 네이버 브라우저 세션을 오래 안정적으로 유지하기에는 적합하지 않다. 공식 문서상 schedule은 cron 기반이며 최단 5분 간격으로 실행 가능하지만, 실제 네이버 브라우저 자동화의 로그인 상태와 캡차 대응에는 한계가 있다. 권장 용도는 CI, 테스트, CSV 검증, 백업 점검이다.

## 9. 단계별 개선 계획

### Phase 0: 기준선 고정

- 현재 변경사항을 별도 브랜치로 정리.
- Python 의존성 설치/테스트 환경 복구.
- `requirements-dev.txt`에 `pytest` 추가.
- `modules/automation/__init__.py` eager import 제거.
- Next SWC optional package 재설치 또는 `npm ci` 기준 복구.

### Phase 1: 블로그 전용 경량 모드

- `AUTOBLOG_MODE=naver_lite` 추가.
- 멀티채널, VLM 경쟁, 트렌드 자동수집을 기본 비활성화.
- CSV 입력 → job 생성 API/CLI 추가.
- 하루 목표 기본값 3으로 조정.

### Phase 2: 페르소나/글 품질 재설계

- 첨부 페르소나를 기본 프로필로 반영.
- `카페사장 아빠` 중심 프롬프트를 `윤서재` 중심으로 재작성.
- 투자 초심자 대상 글 구조와 투자 리스크 문구 추가.
- 제목-본문 일치, 한국어 혼입, 금지 표현, 일반론 과잉을 자동 검사.

### Phase 3: 텔레그램 승인 플로우

- `draft_pending_approval` 상태 추가.
- 초안 생성 후 텔레그램으로 본문 전송.
- 승인 버튼 callback을 job 상태 전환에 연결.
- 수정 요청 메시지를 다음 재작성 프롬프트에 반영.

### Phase 4: 네이버 임시저장 모드

- `BasePublisher`에 `save_draft()` 추가.
- `PlaywrightPublisher.save_draft()` 구현.
- 임시저장 버튼 셀렉터와 성공 검증 스크린샷 저장.
- 기존 `publish()`는 최종 발행 전용으로 유지.

### Phase 5: 이미지 품질 안정화

- 글당 이미지 슬롯을 2~3개로 제한.
- AI 이미지/스톡 이미지 비율을 토픽별 고정.
- 이미지 배치 전 텍스트 흐름 검사.
- 이미지 출처, AI 활용 설정, 파일 크기, 업로드 성공 로그 저장.

### Phase 6: 배포 및 운영

- 로컬 LaunchAgent로 1주일 안정화.
- 이후 Cloud Run Jobs로 생성/승인 서버를 분리.
- 네이버 임시저장은 로컬 브라우저 워커 또는 Cloud Run headless 중 성공률 높은 쪽 선택.
- 월 비용 대시보드: LLM 호출 수, 이미지 호출 수, 실패 재시도 비용 표시.

## 10. 성공 기준

- CSV 30개 주제 기준, 90% 이상이 품질 게이트 통과.
- 준비 초안 10개 중 0개가 제목-본문 불일치.
- 준비 초안 10개 중 0개가 외국어/깨진 표현 혼입.
- 텔레그램 승인 버튼으로 상태 전환 성공률 95% 이상.
- 네이버 임시저장 10회 연속 성공.
- 하루 3개 기준 월 LLM/이미지 비용 3달러 이하 유지.
- 캡차/로그인 만료 발생 시 자동 중단 후 텔레그램 알림.

## 11. 우선순위 제안

가장 먼저 할 일은 기능 추가가 아니라 "발행을 멈추고 임시저장 중심으로 안전장치를 바꾸는 것"이다. 그다음 글 품질을 윤서재 페르소나에 맞게 고정해야 한다.

추천 순서:

1. 의존성/빌드/테스트 기준선 복구.
2. `publish()`와 `save_draft()` 분리.
3. 텔레그램 초안 승인 상태 추가.
4. 페르소나 프롬프트 교체.
5. CSV 입력과 topic memory 기반 연계성 강화.
6. 로컬 1주일 운영 후 Cloud Run 하이브리드 배포.
